# Zeratul

星际酒馆 PVP 模式的泽拉图预言牌推理工具，使用原生 `HTML + CSS + JavaScript` 实现，无框架、无构建链路、无后端依赖。

## 项目简介

本项目用于辅助玩家记录"进场卡牌"和"是否获得隐刀"的反馈，并基于泽拉图技能规则对核心包中的预言牌进行排除、收敛和推荐下一步试探卡牌。

当前实现是一个纯静态前端页面（V2.0.1）：

- 页面入口是 `index.html`
- 业务逻辑集中在 `app.js`
- 样式集中在 `styles.css`
- 卡牌数据以多个独立脚本文件形式放在 `resource/data/`

## 已实现功能

### 1. 卡包加载与数据管理

- 启动时默认加载核心包 `core`
- 额外卡包通过左侧复选框启用/停用
- 卡牌数据通过动态插入 `<script>` 标签异步加载到 `window._packData`，已加载的脚本会缓存避免重复注入
- 切换卡包后会清空当前猜测牌局记录并重新计算候选牌
- 当前代码中 `MAX_EXPANSION_PACKS = 100`，因此实际上没有启用"最多选择两个扩展包"的限制

### 2. 进场卡牌筛选与录入

- 按种族筛选可选卡牌（星灵、异虫、人族、中立）
- 按等级筛选可选卡牌（等级按钮根据当前种族动态生成）
- 展示符合条件的卡牌按钮，点击后作为本次进场对象
- 通过"奇数 / 偶数或0"记录本次是否获得隐刀
- 已经进场过的卡牌不可重复录入（按钮置灰并划线）
- 进场按钮带点击闪光动画反馈

### 3. 历史记录管理

- 表格展示每次进场记录
- 展示字段包括序号、卡牌名、种族、等级、单位数、价值、是否获得隐刀
- 支持删除单条记录
- 支持清空全部记录（需确认）

### 4. 候选预言牌计算

- 仅从核心包卡牌中筛选预言牌候选项
- 依据每条历史记录校验候选牌是否一致
- "接近"判定规则为以下任一条件成立：
  - 种族相同
  - 初始单位数相同
  - 价值差值小于等于 `200`
- 支持按种族、单位数、价值排序展示候选牌
- 显示候选牌数量统计（当前候选数 / 核心包总数）

### 5. 约束推断

- 从"奇数"记录中提取正向约束（close cards）
- 从"偶数 / 0"记录中提取排除约束（not close cards）
- 对种族、单位数、价值三个维度做约束传播（最多迭代 10 轮收敛）
- 在界面中将每个维度标记为：
  - 已锁定（locked）— 已确定唯一值
  - 已缩小（narrowed）— 已排除部分可能
  - 未知（unknown）— 暂无信息
- 约束面板根据状态显示不同提示信息

### 6. 匹配维度与优先级

- 为每张候选牌计算优先级分数（0~5 分），基于种族、单位数、价值三个维度的匹配情况
- 引入稀有度加权：匹配维度在候选集中占比越低，得分越高
- 候选牌表格中以彩色标签显示匹配维度（种族 / 单位数 / 价值）
- 与已知 close 记录匹配的候选行高亮显示

### 7. 精准预测与下一步推荐

- 根据当前候选集计算收敛进度百分比，并以进度条展示
- 估算还需多少步可以收敛到唯一候选（基于 log2 计算）
- 对未使用卡牌计算信息增益（Shannon 熵），推荐最适合继续试探的 Top 5 卡牌
- 每张推荐卡显示信息增益百分比、奇数/偶数各剩余候选数
- 推荐牌带"选用"按钮，点击可自动跳转到对应种族和等级并选中该卡
- 支持按等级过滤推荐池（多选按钮）
- 当候选只剩一张时直接显示已锁定的预言牌（紫色高亮卡片）
- 当无候选牌时显示警告提示

### 8. 预言确认与新一轮开始

- 在候选列表中点击"预言牌"按钮可打开确认弹窗
- 弹窗带滑入动画、毛玻璃背景遮罩
- 确认后清空当前记录，开始下一轮推理
- 支持取消、关闭按钮、点击遮罩层关闭

### 9. 界面与交互

- 固定顶部标题栏，包含 Logo 和外链导航（酒馆共享黑名单、工具说明、@阿冰ベ）
- 三栏布局：卡包选择区（左侧）/ 操作与历史记录区（中部）/ 精准预测与候选牌区（右侧）
- 星际争霸2 神族风格的配色方案（深空蓝底色、青色主调、金色高亮、紫色预言）
- 边框纹理使用 `border-image` 实现星际风格边框
- 背景图和纹理底图营造沉浸感
- 自定义 SC2 风格滚动条（支持 Webkit 和 Firefox）
- 包含帮助提示气泡（问号图标悬停显示规则说明）
- 按钮高亮、选中态、禁用态样式完整
- 已适配平板和手机宽度，`768px`（三栏变单栏）与 `480px`（进一步缩小间距和字号）处有响应式断点

## 模块说明

### `index.html`

负责页面结构：

- 顶部标题栏（Logo、标题 V2.0.1、外部链接）
- 左侧卡包选择区（`aside.sidebar`）
- 中部进场控制区和历史记录表格（`div.main-content`）
- 右侧精准预测面板和候选牌表格（`aside.sidebar-right`）
- 预言确认模态框（`div#prophecyModal`）

### `app.js`

负责全部业务逻辑，采用 IIFE 组织，主要包含以下部分：

- **常量与卡包注册表** — `CLOSE_THRESHOLD`（200）、`MAX_EXPANSION_PACKS`（100）、`RACE_CN` 种族中英映射、`PACK_REGISTRY` 卡包列表
- **动态加载卡包脚本** — `loadPackScript()` 通过 Promise + `<script>` 标签异步加载
- **全局应用状态** — `App.state` 包含 cards、guesses、sortBy、selectedRace、selectedLevel、selectedCardId、enabledPacks、predictionLevels
- **DOM 缓存与事件绑定** — `cacheEls()`、`bindEvents()`
- **猜牌录入、删除、重置逻辑** — `addGuess()`、`removeGuess()`、`confirmProphecy()`
- **候选牌一致性判定** — `isClose()`、`candidateConsistent()`、`getCandidates()`
- **约束推断与优先级计算** — `buildConstraints()`（含排除集构建、约束传播、候选分布统计、维度摘要）、`calcPriority()`（含稀有度加权）
- **推荐牌信息增益计算** — `calcRecommendations()`（Shannon 熵排序，取 Top 5）
- **各区域渲染逻辑** — `renderAll()`、`renderPackToggles()`、`renderRaceLevelSelectors()`、`renderCardButtons()`、`renderHistory()`、`renderCandidates()`、`renderConstraintSummary()`、`renderPrediction()`

当前状态持久化接口 `persist()` 与 `restoreState()` 已预留，但尚未实现。

### `styles.css`

负责全部视觉样式与响应式布局：

- **主题变量定义** — 深空底色、面板色、文字色、神族青色调、金色、预言紫、危险红、边框色、间距与圆角
- **三栏布局** — 左侧 180px 固定侧栏、中部 650px 固定宽度、右侧弹性填充
- **卡牌筛选按钮** — 种族/等级/卡牌按钮样式、选中态、已用态
- **反馈按钮** — 奇数/偶数选择按钮样式
- **候选牌标签** — 匹配维度彩色标签（种族青、单位数金、价值紫）
- **约束摘要面板** — 锁定/缩小/未知三种状态标签样式
- **精准预测面板** — 进度条、推荐卡片列表、信息增益展示、锁定状态紫色卡片
- **表格布局** — 历史表格（flex 撑满高度可滚动）、候选牌表格（自然撑开）、列宽分配
- **模态框** — 毛玻璃背景、border-image 星际边框、滑入动画
- **问号提示气泡** — 悬停显示的帮助文本
- **自定义滚动条** — SC2 风格渐变滚动条
- **按钮动画** — 进场按钮点击闪光（`clickFlash` keyframe）
- **响应式断点** — 768px 平板适配（三栏变单栏）、480px 手机适配

### `resource/data/*.js`

负责卡牌数据注册。每个文件都会向全局对象写入如下结构：

```js
window._packData["packKey"] = {
  name: "卡包名",
  cards: [
    { id, race, level, number, value }
  ]
};
```

## 构建与运行方式

本项目没有打包、编译、测试或发布脚本，属于直接运行的静态页面项目。

### 本地运行

可直接使用任一静态文件服务器打开项目根目录，例如：

```powershell
cd D:\Works\Zeratul
python -m http.server 8080
```

然后访问 `http://localhost:8080`。

如果浏览器允许，也可以直接打开 `index.html`，但由于项目通过脚本标签动态加载数据文件，更推荐通过本地静态服务器访问。

### 构建方式

- 无 `npm` / `pnpm` / `yarn`
- 无 `package.json`
- 无 bundler
- 无转译流程
- 无 CI 配置

也就是说，当前"构建"方式就是直接托管静态文件。
